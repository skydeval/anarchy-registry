After building the goodgirls registry, I wanted to try something more ambitious: pride theme backgrounds that changed on each page load, customization because autonomy is a large part, so I made a mechanism for anyone who uses it to also add two more handles if they want, a way to change them, and delete them too. That's something I'd never seen in handle generators like this — a way to change and/or delete the handle. So I added that. And I shaped the whole project to be able to function on its own, without anyone needing to monitor it or keep it working. Even if no one ever logs into the admin section, it would still be able to function. That's how I designed it, and for seven months it worked.

Part of this project, migrating it into Rust, was not only to accomplish that, but to show how much I've grown these last seven months through my coding adventures. So each aspect I tweaked to improve: the themes, I added variances that gave users the means to change the direction of the theme, and manually select a particular one if desired. I added security measures. Seven months ago I hadn't developed a non-enumeration standard, which I've since adopted as almost part of what I aim to implement in personal projects, so I applied that here too. If someone isn't authorized to access it, they don't get an error code — they just get a 404. I'm the only admin, so if anyone tries to hack in... good luck.

The adversarial review process didn't surface many surprises — I've been using adversarial refinement as a standing practice on my larger projects for months, so the instinct to attack an idea before building it was already there by the time I wrote this design doc.
